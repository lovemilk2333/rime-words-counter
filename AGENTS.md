# 统计与多端同步需求说明书 (Word Counter Target Specification)

## 1. 基础环境与配置管理
* **应用标识**：定义全局应用名称为 `milk-word-counter`。
* **路径自适应**：跨平台（Windows, macOS, Linux）自动识别路径分隔符、环境变量与系统的配置目录（如类 Unix 系统下的 `XDG_CONFIG_HOME` 或 `~/.config`），所有生成的本地文件及同步文件夹均存放在此配置根路径（以下统称 `$config_path`）下。
* **配置文件 (`$config_path/config.json`)**：
  * **唯一设备标识**：使用 **UUID v4** 标准格式作为 `machine_id` 自动写入配置文件。如果配置缺失或初次运行，则动态生成并持久化，后续始终从中读取。
  * **同步路径动态解析**：支持定义 `"sync_dir": "$config_path/sync"` 占位符格式。在脚本启动时，系统需要将 `$config_path` 动态替换为当前的绝对路径。若用户在 JSON 中直接填写了不带占位符的绝对路径，则直接使用用户的绝对路径。
  * **阈值控制**：
    * `commit_threshold`：单次触发导出的用户提交上屏次数阈值（默认 `32`）。
    * `history_threshold`：闲时清理历史文件时的最大保留快照数量阈值（默认 `16`）。
  * **多维度权重正则配置 (`count_rates`)**：
    支持用户通过键值对灵活配置不同文本类别的统计权重速率。例如：
    ```json
    "count_rates": {
        "cjk": 1,
        "ascii": 0.33,
        "<正则表达式>": 2
    }
    ```
  * **时间分片配置 (`state_split`)**：
    按时间粒度分片存储统计数据，便于查看历史趋势。支持的时间粒度：`Nhour[s]`、`Nday[s]`、`Nweek[s]`、`Nmonth[s]`、`Nyear[s]`。例如：
    ```json
    "state_split": {
        "monthly": "1month",
        "daily": "1day"
    }
    ```
    每个分片存储在 `$config_path/<分片名>/` 目录下，文件名为对应的时间段标识。
  * **分片保留期限 (`state_retention`)**：
    控制各分片的历史数据保留时长，超期文件在闲时同步时自动清理。例如：
    ```json
    "state_retention": {
        "monthly": "24months",
        "daily": "90days"
    }
    ```

## 2. 基于频数正则的文本切片统计机制
当用户触发文本上屏时，脚本需要根据配置文件中的 `count_rates` 对文本进行多维度识别：
* **匹配优先级**：优先匹配用户自定义的字符串 `<正则表达式>`。凡是被自定义正则拦截提取的文本片段，必须从当前待统计的流中完全抹除，防止其在后续的通用规则中被重复计算。
* **通用规则识别**：
  * `cjk`：准确统计通过正则拦截后剩余文本中的中日韩统一表意文字（汉字）个数。
  * `ascii`：准确统计通过正则拦截后剩余文本中的标准 ASCII 可视及空白字符个数。
* **本地状态就地加权变动 (`$config_path/state.json`)**：
  上屏文本统计完成后，需要立刻在本地计算各项分类的**加权数值**（`原始匹配个数 * 配置权重`），并将其就地合并更新到状态文件的 `counts` 字典中。同时，动态重新计算并维护全量加权总和：
  $$\text{total} = \sum (\text{counts.values()})$$

## 3. 退格检测与错误率统计
脚本通过 `lua_processor` 拦截按键事件，统计字符删除次数（错误率）：
* **preedit 内删除**：输入过程中未确认的字符删除，通过追踪 preedit 长度变化精确统计删除的字符数。
* **已提交文本删除**：光标左移删除已确认的文本，近似统计（每次退格计 1 次）。
* **错误数据存储**：错误计数存储在 `state.json` 的 `errors` 字段中，并同步写入各时间分片文件。

## 4. 时间分片存储
按配置的 `state_split` 规则，将统计数据按时间粒度分片存储：
* **分片文件结构**：
  ```json
  {
      "period": "2025-06",
      "total": 1234.56,
      "counts": { "cjk": 1000, "ascii": 234.56 },
      "errors": 42,
      "commits": 156
  }
  ```
* **存储路径**：`$config_path/<分片名>/<时间段标识>.json`
* **清理机制**：根据 `state_retention` 配置，在闲时同步时自动清理过期分片文件。

## 5. 数据缓冲区与同步导出 (生成 33 行模式)
* 当用户输入提交次数达到配置文件设定的阈值（`commit_threshold`，默认 `32`）时，立即清空当前的内存缓冲区，并向配置的 `sync_dir` 目录下导出一个带有当前设备 `machine_id` 和时间戳的 `.jsonl` 文件。
* **文件行数格式**：
  * **前 32 行**：每行必须为一个独立的标准 JSON 字符串，内容包含当前行的时间戳、该次提交各分类的**原始匹配个数（Raw Counts）** 以及本机的 `machine_id`。
  * **第 33 行（尾行）**：必须写入一行用于防数据丢包和断电截断校验的纯文本尾封，固定格式为：
    ```text
    --$app_name--done--$machine_id
    ```

## 6. 闲时双向异步合并与滑动窗口清理
脚本需要提供一个被输入法闲时或初始化触发的异步任务，用于消费和对齐 `sync_dir` 下的其他节点数据：
* **单次快照读取**：在开始执行同步时，一次性获取当前 `sync_dir/*.jsonl` 的文件列表，读取过程中不再更新此快照。
* **尾封断裂校验**：依次读取文件时，必须严格检查文件的最后一行。若末尾没有标准的 `--$app_name--done--$machine_id` 尾封，说明该文件在传输或写入中途遭遇损坏，则直接**忽略并跳过**该文件。
* **环回过滤机制**：提取尾封中的 `$machine_id`。如果该文件是由**当前设备自身**导出的，则不将其数据重复合并进本地词数，但直接将其文件名标记为已处理。
* **跨端求和合并**：对于验证合法且来自其他设备的文件，将其内部前 32 行每行对应的原始频数提取出来，按照**当前设备本地配置的权重速率**进行折算，并合并累加到 `$config_path/state.json` 的 `counts` 字典和 `total` 总和中。
* **文件名幂等性防护**：所有处理过的文件名（不论是本机导出的还是合并成功的）都必须存入 `state.json` 的 `processed_files` 状态表。再次启动同步时，直接跳过已记录的文件名，防止多端循环解析导致的计数无限重叠。
* **滑动窗口垃圾清理**：统计已在状态表中记录、处理完毕的旧 `jsonl` 文件名称，将它们按时间先后排序。当总数超过 `history_threshold`（默认 `16`）时，自动尝试从磁盘中物理删除排在最前面的 $\lfloor\text{threshold} / 2\rfloor$（默认 `8`）个最老的数据文件，以实现磁盘空间的滑动窗口循环复用。

## 7. 多设备状态同步与 JSONL 生命周期管理
脚本在 `sync_dir` 中维护每个设备的同步状态文件，实现跨设备的处理进度追踪与 JSONL 文件的自动清理：

### 7.1 设备同步状态文件 (`dev_$machine_id.json`)
* 每个设备在 `sync_dir` 下维护一个以统一前缀 `dev_` 开头的状态文件，格式为 `dev_$machine_id.json`。
* **文件内容**：
  ```json
  {
      "machine_id": "设备 UUID",
      "last_sync": "最后同步时间戳",
      "processed_files": {
          "其他设备的 jsonl 文件名": true
      },
      "exported_files": {
          "本机导出的 jsonl 文件名": true
      }
  }
  ```
* **`processed_files`**：记录本机已成功处理（合并加权数据）的所有 JSONL 文件名，用于幂等性防护。
* **`exported_files`**：记录本机导出的所有 JSONL 文件名，用于追踪全端处理进度。

### 7.2 导出时注册
当 `emit_sync_file()` 生成一个 JSONL 文件后，立即将该文件名写入本机 `dev_$machine_id.json` 的 `exported_files` 字典中。

### 7.3 闲时同步与全端删除
`sync_idle_job()` 在每次执行时：
1. **设备发现**：扫描 `sync_dir/dev_*.json`，读取所有已知设备的同步状态。
2. **JSONL 处理**：遍历 `sync_dir/*.jsonl` 文件，对未处理的文件执行尾封校验、环回过滤、跨端加权合并，并更新本机 `state.json` 和 `dev_$machine_id.json` 的 `processed_files`。
3. **全端删除判定**：遍历本机 `dev_$machine_id.json` 的 `exported_files`，对于每个文件，检查所有**其他**已知设备的 `dev_$machine_id.json` 中的 `processed_files` 是否均已包含该文件名。若全部已处理，则从磁盘删除该 JSONL 文件，并从本机 `exported_files` 中移除。
4. **滑动窗口清理**：对本机 `state.json` 的 `processed_files` 执行超阈值清理（同第 4 节规则）。
